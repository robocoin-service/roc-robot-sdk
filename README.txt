ROC Robot SDK Core

Linux TPM robot recommended one-step install:

curl -fsSL https://raw.githubusercontent.com/robocoin-service/roc-robot-sdk/main/install.sh | bash -s -- <sdkBindingToken>

Manual command after clone:

./roc-robot-tpm-sdk.sh bind <sdkBindingToken>

The SDK package is device-neutral. The binding token is robot-specific and generated by the platform after robot registration.
The SDK reads TPM public key data, asks the TPM private key to sign the challenge payload, and submits the signed report to roc-server.
After binding succeeds, the installer creates and starts a systemd service named roc-robot-agent. The background agent sends a TPM-signed heartbeat every 30 seconds and waits for platform stage verification challenges.
The TPM private key is non-exportable and will not be written to the database.

Service commands:
sudo systemctl status roc-robot-agent
sudo systemctl restart roc-robot-agent
sudo journalctl -u roc-robot-agent -f

Diagnostics are run automatically at the end of install. Manual diagnostics for troubleshooting:
cd ~/roc-robot-sdk && ./roc-robot-tpm-sdk.sh doctor

Architecture:
- SDK Core provides Linux installation, identity binding, TPM/software-RSA signing, heartbeat, challenge verification, diagnostics and standard job/incident contracts.
- Hardware control is implemented by optional adapters. Core does not assume any robot brand, ROS, Unitree SDK or motion interface.
- Third-party developers should implement an adapter for their own robot and declare only the capabilities it can safely execute. See ADAPTER-CONTRACT.md.

Core-only installation:
- The default installation is Core-only and does not detect, build, start or repair any vendor hardware service.
- A normal Linux VM can validate Core installation with ROC_SDK_TEST_MODE=1, ROC_ALLOW_SOFTWARE_IDENTITY=1 and ROC_ADAPTERS=none.

Optional Go2 adapter:
- Install explicitly with ROC_ADAPTERS=go2 before running install.sh.
- The PC reaches the bridge through the robot WiFi address (for example, 172.16.x.x:8080).
- On an onboard Go2/Orin computer, Unitree DDS actions use the internal 192.168.123.x interface, normally eth0.
- The Go2 adapter detects the route to the Go2 controller at 192.168.123.161 and builds the C++ action helper only when explicitly requested.
- Patrol actions run forward, turn, and forward again in one C++ control session; STOP terminates the entire active patrol.
- Long forward demo actions support MOVE_FORWARD/FORWARD up to 60 m; long motion starts asynchronously, reports live motion status, and can be interrupted through STOP.
- Long forward motion defaults to 0.60 m/s, applies an 800 ms acceleration/deceleration ramp, and clamps configured speed to 0.80 m/s.
- LiDAR safety navigation is exposed only through NAVIGATE_FORWARD_AVOID, MOVE_FORWARD_AVOID, or SMART_FORWARD. Existing MOVE_FORWARD behavior is unchanged.
- LiDAR navigation subscribes to rt/utlidar/range_info and rt/sportmodestate, stops on stale sensors or stalled odometry, compares left/right clearance, performs a bounded lateral bypass, rejoins the original route, and stops after repeated failures.
- LiDAR navigation defaults to 0.22 m/s and can be tuned with ROC_GO2_LIDAR_NAV_SPEED, ROC_GO2_LIDAR_STOP_DISTANCE, ROC_GO2_LIDAR_SIDE_CLEARANCE, ROC_GO2_LIDAR_LATERAL_SHIFT, and ROC_GO2_LIDAR_MAX_BYPASS_ATTEMPTS.
- The built-in VideoClient is available for later semantic camera recognition. Physical clearance and stop decisions remain LiDAR-based and require no additional camera hardware.
- Read-only hardware checks: go2_helper/build/go2_lidar_navigator eth0 --probe and go2_helper/build/go2_camera_probe eth0. Neither command claims motion control.
- Start a bounded LiDAR route through the local bridge with action NAVIGATE_FORWARD_AVOID and meters set to the requested forward distance. The existing /api/v1/cancel endpoint remains the emergency stop path.
- Set ROC_GO2_NETWORK_INTERFACE only when automatic detection is not correct.
